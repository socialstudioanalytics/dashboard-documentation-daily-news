SSA platform — top-level flow & Pipeline C internals

Two reference diagrams with accompanying notes. Generated 2026-06-16. Scope: data-app ↔ client architecture and the Pipeline C stage chain. Figures are self-contained and theme-independent; surrounding prose inherits the host docsite styling.

1. Platform overview

Upstream sources feed the data app via S3. The data app exposes two operator surfaces and runs both pipelines in parallel, then crosses a one-way contract boundary into the client app. The constraints that are easy to lose at this altitude — the second operator surface, the separate raw_posts path, the additive webhook budget, and the admin-only status of Pipeline C — are drawn explicitly rather than collapsed.

Accompanying points

  • The two converging lines under the pipelines reflect the real contract: both pipelines emit per-day reports against the same webhook endpoint, with Pipeline C's fields layered additively on top of the Pipeline B shape rather than substituting them. The client renders either payload without knowing which produced it.
  • Today only Pipeline B reaches end users. Pipeline C output is admin-only, gated behind the Plans A/B/C convergence work.
  • raw_posts sync is a distinct path on its own cadence (currently Telegram-only) with its own kill-switch; Pipeline C does not depend on it for its corpus (Stage 1 reads S3 directly), but the client daily-summary endpoint does.
  • The contract is one-directional: data flows down only. The client never asks the data app to re-process, re-classify, or add a field — any such need is scoped and escalated separately.
  • The webhook payload is measured against a hard ~1 MB proxy ceiling, which drives the trim and byte-budget policy.

2. Pipeline C internals

The stage chain that Figure 1 collapses into a single purple box, with the known gaps flagged where they actually occur. The headline is domain routing: the anchor-centroid gate is near-dormant and the LLM fallback does essentially all the disambiguation.

Accompanying points

  • Stage-order drift between two canonical docs. The route-before-merge ordering shown here follows the code-grounded architecture doc, where domain routing is Stage 4 and the Jaccard merge is Stage 5b. The narrative system-reference lists them the other way round — merge before routing. This is a live discrepancy to reconcile against the orchestrator code before anyone relies on the order.
  • unclassified is a failure sentinel, not a domain. It is the LLM-failure-only sentinel (ADR 0007), set when the fallback returns an invalid response — not a first-class cross-lens domain.
  • Eight prompts, not nine. The shipped set is eight (seven canonical domains plus the unclassified fallback), enforced by the assertion in pipeline_c_prompts.py, even though the count is sometimes referred to as nine.
  • Routing reality. The anchor-centroid gate fires on roughly 0.2% of clusters; the gpt-4.1-mini fallback performs nearly all domain disambiguation. This is either intentional tight calibration or an anchor-drift signal — it has not been adjudicated.
  • Merge dormant. The Stage 5b Jaccard merge folded zero clusters across the 14 days ending 2026-05-13 (confounded by the sub-floor coverage period). It remains a safety net, not a load-bearing quality lever.
  • Prompts are starter drafts. The eight per-domain prompts and the synthesis prompt that ship today are Replit-authored starter drafts pending the analyst-revision pass.
