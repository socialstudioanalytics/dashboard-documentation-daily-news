Phase 1 — Immediate fixes (do first, mostly quick)
Remove hardcoded date rangesdashboard_router.py · embedding sender
2 / 7 done		 Critical Low1–2 days Before Calendar dates are baked into the code, so it keeps analyzing the same old days unless edited every run — silently producing outdated reports. dashboard_router.py is fixed to DATE_FROM = "2026-04-13"; run routing today and today’s posts never reach the dashboard.
Move secrets to a secure vault & rotate.env → Secret Manager
0 / 7 done		 Critical Low1–2 days Before Keys are stored in plain text in the project files; anyone opening the folder can use them. They belong in a vault, and exposed ones must be replaced. Each service ships a plaintext .env (OpenAI key, Mongo URI); live Telegram session strings were even shared in chat.
Replace default DB password & close admin paneldocker-compose · Mongo-Express
0 / 7 done		 Critical Low<1 day Before A well-known default password protects the DB, and an admin panel is reachable with it — an open door to all data. docker-compose.yml uses adminpassword123 and exposes Mongo-Express on :9081 with the same login.
De-duplicate posts before AI analysissimilarity → NLP + cache
4 / 7 done		 High Medium~1 week Before The AI is paid to read near-identical posts repeatedly. Filtering duplicates first is the single biggest cost saving available. A press release reposted across 40 accounts is sent to OpenAI individually because similarity runs after NLP.
Add database indexesposts: status, date, section
0 / 6 done		 High Low1–2 days Before No quick-lookup shortcuts for the filtered fields, so the DB scans everything — slower and pricier as data grows. The NLP poller scans the growing posts collection each cycle because the analysis-status fields aren’t indexed.
Standardize the database nameunify DATABASE_NAME
0 / 6 done		 High Low<1 day Before Different parts assume different DB names, so data can quietly end up in the wrong place and look “missing.” similarity_engine.py defaults to socialscraper while others use fastapi_db/socialstudio.
Phase 2 — Stability & cost control
Consolidate manual scripts into one toolsingle CLI + shared lib
0 / 7 done		 High Medium1–2 weeks Before Many hand-launched scripts; one person knows which to run. One clear tool removes that single-person dependency. Backfilling means choosing among reprocess_all_posts_openai.py, run_similarity_and_routing_once.py, etc., each with its own MongoClient.
Switch always-on servers to scheduled jobsCloud Run Jobs + Scheduler
0 / 7 done		 High Medium~1 week After Servers run 24/7 even when idle, so you pay for nothing. On GCP they run only when needed. ss-data-scraper & ss-data-nlp run as restart: unless-stopped polling all night.
Use a pay-per-use databaseFirestore (MongoDB-compatible)
0 / 7 done		 High Medium~1 week After Pay only for what’s used instead of renting a DB around the clock — big saving for bursty workloads. posts & dashboard_* live in an always-on Mongo that bills through the quiet hours.
Add monitoring, alerts & cost dashboardCloud Monitoring + budgets
0 / 6 done		 High Medium~1 week After Nothing warns when the pipeline stalls or spend spikes. Dashboards + alerts catch problems before they cost money. Only loguru/print logs exist; a stalled poller or exhausted OpenAI keys go unnoticed until the dashboard stops updating.
Set up backups & recovery planbackup + tested restore + DR
0 / 6 done		 High Medium~1 week After No tested copy to fall back on if data is deleted or corrupted. A verified backup makes the business recoverable. All posts & dashboard_* live in one Mongo with no documented backup.
Harden container & access permissionsnon-root, least-privilege
0 / 7 done		 Medium Low–Med2–4 days After Apps run with maximum privileges and a dev setting left on. Tightening limits the blast radius of any compromise. ss-core-api runs uvicorn … --reload as root; only ss-data-nlp uses a non-root user.
Phase 3 — Scalability (toward 100k posts/day)
Add a work queue between stagesPub/Sub + parallel workers
0 / 7 done		 High High1–2 months After Stages hand off rigidly, so a surge jams the line. A queue buffers work and lets workers share the load. Stages coordinate via status flags in posts; a hot-window burst floods it faster than the single NLP poller drains it.
Run the heavy “storyline” analysis separatelyon-demand GPU batch
0 / 7 done		 High Medium1–2 weeks After The heaviest step runs glued to everything else and can crash the service. It should run on its own, only when needed. embedding_stage.py loads e5-large + UMAP + HDBSCAN inside the same service that does per-post enrichment.
Send AI requests in batchesasync, rate-limited OpenAI pool
0 / 7 done		 High Medium~1 week After AI requests go one at a time — slow and pricier. Grouping and parallelizing speeds processing and lowers cost. analyze_post() is called once per post in the loop; keys are rotated one at a time, so backlogs run serially.
Phase 4 — Productization (sell, operate, scale to clients)
Add automated tests & release checksunit tests + CI gates
0 / 6 done		 High Medium2–3 weeks After No automatic safety checks, so a small change can silently break something until a client notices. A wrong edit to SECTION_TO_COLLECTION in dashboard_router.py could misroute posts with nothing to catch it.
Data governance: retention & compliancePII, retention, ToS
0 / 7 done		 High Mediumongoing After No rules for what personal data is collected, how long it’s kept, or the legal basis — a real compliance/reputation risk. Scraped posts, account profiles, and Telegram session credentials are stored indefinitely with no retention policy.
Add backup providers for key servicesLLM / scraper fallback
0 / 7 done		 Medium Medium2–3 weeks After Total reliance on a few vendors. If one raises prices or cuts access, operations stop. Alternatives keep the service running. Enrichment calls only OpenAI; the flowchart lists Gemini as an alternative but no code path uses it.
